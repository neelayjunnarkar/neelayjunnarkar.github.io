Pathfinder
Posted on October 24, 2016 by Neelay Junnarkar
A C++ implementation of A*, Dijstra's algorithm, and breadth-first search. Show more...

Well, after creating a maze generator, I clearly had to make a program to solve the mazes. So here it is.

I implemented several path finding algorithms including breath-first search, Dijkstra's algorithm, and A*. The breadth-first algorithm is the simplest and simply searches, starting at the start point, adjacent cells until it finds the goal. Dijkstra's algorithm searches the graph (the grid in this case) more intelligenty by prioritizing adjacent cells that have less weight, or effectively can be considered to be 'closer'. A* is an improvement upon Dijkstra's and uses its knowledge of the position of the goal to help increase performance.

Example in a random-weight grid with some obstacles.

Legend:

  • Green Cell - Start
  • Red Cell - Goal
  • Green-tinted Cells - cells that have not been visited
  • Red-tinted Cells - cells that have been visited
  • Weight is represented by shades of gray: the lighter the cell, the less weight it is
  • Black Cells - not nodes in the graph (intraversable)
  • Blue Small Squares - Path

And now here is an example of the pathfinder operating on a 51x51 maze. Same legend as before (but all cells are equally weighted in the maze).

Maze Generator
Posted on October 21, 2016 by Neelay Junnarkar
A maze generator in C++. Show more...

This maze generator uses the backtracking algorithm in which a head node goes out from the starting cell in a random path until it hits a dead end. At this point, the head node back tracks until it reaches a cell from which it can create a viable offshoot from the main path. I am considering modifying my program so there are several head nodes, and thus several paths being created at the same time. This would address the problem of the backtracking algorithm that it tends to create simple long chains with easily identifiable offshoots.

Example 101x101 maze generated. Green is the starting cell (top right ish) and red is the goal cell (top left ish). Starting and ending points are randomized, but are guaranteed to be the pair of points with the largest distance (not displacement) between them.

Conways Game of Life
Posted on October 19, 2016 by Neelay Junnarkar
A C++ implementation of Conway's Game of Life. Show more...

Conway's Game of Life is a simulation of interacting cells. Each cell in the grid can either be dead or alive, and the state of the cell in the next step in time is determined by 4 rules:

  1. Any live cell with fewer than two live neighbours dies
  2. Any live cell with two or three live neighbours lives on to the next generation
  3. Any live cell with more than three live neighbours dies
  4. Any dead cell with exactly three live neighbours becomes a live cell


One of the interesting parts of my implementation is a vector of functions, in which each function is a rule. This allows for easy customization and hot-swapping of rules.


Some of my friends said I couldn't write Conway's Game of Life in under an hour. I said I would be able to do it in under 45 minutes.

I was wrong, but I still got it in under 1 hour at 57 minutes.

Command Argument Parser
Posted on September 24, 2016 by Neelay Junnarkar
A C++ parser for arguments to main. Show more...

This is a parser I wrote that allows me easily take in environment arguments. I have found a need for this in some of my vision and networking programs.

This single-file header-only library allows simple parsing of 3 types of arguments: boolean, labeled, and positional. For boolean arguments, my parser simply checks whether the flag is passed or not. For labeled arguments, my parser returns the values that come after a flag such as "--coordinates". Position arguments are a simpler version of labeled arguments meant only for the most basic input. For these, my parser returns the values that are at specific indices

Followers
Posted on August 7, 2015 by Neelay Junnarkar
A simple javascript program in which blocks follow the cursor. Show more...

Here's a little thing I was playing around with as I am learning Javascript.

The blocks are randomly sized, positioned, and colored (well, the color is from a set). The speed of each block is a function of the distance between the block and the mouse--the speed is the log of the distance, where each block uses a randomized base for the log. The different bases used in the log create the varying speeds. The blocks angle towards the cursor as well.
